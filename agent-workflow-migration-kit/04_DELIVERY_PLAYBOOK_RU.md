# 04. Playbook приемки, деплоя и smoke

## Когда нужен delivery loop

Если задача меняет runtime behavior, интеграции, UI, background jobs или
инфраструктурное поведение, после merge нужен delivery loop.

## Delivery flow

1. Merge PR в `main`.
2. Синхронизировать локальный `main`.
3. Обновить test/staging environment.
4. Развернуть новую версию.
5. Проверить health-checks.
6. Выполнить smoke test новой фичи.
7. Проверить logs и очереди.
8. Зафиксировать:
   - что проверяли
   - что получилось
   - есть ли follow-up issues

## Что такое хороший smoke test

Smoke test не должен быть абстрактным.
Он должен проверять пользовательский путь end-to-end.

Примеры формулировок:

- новая команда в боте реально доступна и отвечает
- новая фича проходит весь pipeline от входа до результата
- новая настройка сохраняется и влияет на поведение
- новый модуль реально присутствует в финальном выводе

## Что проверять на тестовом сервере

- контейнеры/процессы поднялись
- health endpoint отвечает
- background worker жив
- очередь не залипла
- логи не содержат свежих исключений
- интеграция с внешними сервисами не сломана

## Когда delivery считается завершенным

Только когда:

- deployment завершен
- smoke test выполнен
- фича подтверждена в рантайме
- явных operational проблем нет
