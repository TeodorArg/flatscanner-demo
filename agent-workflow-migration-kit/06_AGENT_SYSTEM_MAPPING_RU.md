# 06. Как адаптировать kit под другую агентную систему

## Проблема

Разные системы агентов по-разному называют роли и по-разному умеют:

- редактировать файлы
- работать с git
- открывать PR
- запускать проверки
- создавать подагентов
- вести долгий orchestration loop

Если не сделать mapping явно, workflow начнет расползаться.

## Правильный способ адаптации

Нужно сопоставить не названия агентов, а их функции.

### Роль 1. Orchestrator

В новой системе должен существовать один агент или один режим работы, который:

- читает repository memory
- формулирует specs
- режет работу на slices
- следит за PR loop
- не считает задачу завершенной до merge-ready

### Роль 2. Implementation Agent

Это агент, который:

- меняет product code
- пишет тесты
- обновляет `tasks.md`
- коммитит bounded changes

### Роль 3. Review Agent

Это агент, который:

- анализирует PR после implementation
- ищет баги, риски, регрессии и несоответствие контракту
- не пишет новую фичу, а проверяет уже написанное

## Если в новой системе нет отдельных агентов

Тогда нужно искусственно разделить роли через prompts:

- один prompt mode для orchestration
- другой prompt mode для implementation
- третий prompt mode для review

Главное:

- не смешивать implementation и review в один и тот же шаг без явного переключения роли

## Если в новой системе нет worktrees

Нужно сохранить хотя бы логический аналог:

- отдельная ветка на каждый task slice
- отдельная рабочая директория или отдельная изоляция контекста

Если даже этого нет, migration будет неполной.

## Если в новой системе нет PR API

Тогда workflow можно адаптировать так:

- implementation идет в feature branch
- review проводится по diff/patch
- merge-ready состояние фиксируется вручную

Но нужно сохранить саму идею quality gates.

## Если в новой системе другая OS и другой shell

Это нормальный сценарий.

Нужно сопоставить:

- automation scripts старой среды
- с нативной automation-обвязкой новой среды

Например:

- `*.ps1` -> `bash`/`zsh` scripts
- `*.ps1` -> `python -m tools...`
- `*.ps1` -> `make <target>`
- `*.ps1` -> `just <recipe>`

Новая система должна переписать scripts под свою среду, а не копировать их
дословно.

Главный критерий:

- сохраняются те же шаги процесса
- сохраняются те же quality gates
- сохраняется та же изоляция задач
- сохраняется тот же merge-ready contract

## Минимальный mapping checklist

- Кто играет роль orchestrator?
- Кто играет роль implementation agent?
- Кто играет роль review agent?
- Где хранится durable memory?
- Где хранится feature memory?
- Как создается изолированный task context?
- Как открывается или имитируется PR?
- Где живут required checks?
- Что означает merge-ready?
- Как выполняется post-merge deploy/smoke?
